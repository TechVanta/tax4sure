## 1. SECURITY ISSUES

### CRITICAL

| # | Issue | File(s) | Details |
|---|---|---|---|
| S1 | **CORS wildcard on Lambda Function URL** | `terraform/lambda.tf` | `allow_origins = ["*"]` means any website can call your API. Restrict to your S3 website domain. |
| S2 | **CORS wildcard on Documents S3 bucket** | `terraform/s3.tf` | `allowed_origins = ["*"]` on the documents bucket CORS rule. Restrict to your website domain. |
| S3 | **JWT token stored in localStorage** | `lib/auth-client.ts` | localStorage is accessible to any JS on the page (XSS risk). Consider `httpOnly` cookies instead, or accept the risk for a static SPA architecture. |
| S4 | **30-day JWT expiry with no refresh/revocation** | `lambda/lib/jwt.ts` | Tokens are valid for 30 days with no way to revoke them. If a token is leaked, it's valid for the entire period. Consider shorter expiry + refresh tokens. |
| S5 | **FRONTEND_URL response header defaults to `*`** | `lambda/lib/response.ts` | `Access-Control-Allow-Origin` falls back to `"*"` if `FRONTEND_URL` env var is unset. |

### HIGH

| # | Issue | File(s) | Details |
|---|---|---|---|
| S6 | **IAM access keys output in Terraform state** | `terraform/outputs.tf` | `github_actions_secret_access_key` is stored in Terraform state. Ensure the S3 state bucket is encrypted and access-restricted. Consider GitHub OIDC instead of long-lived IAM keys. |
| S7 | **No rate limiting on auth endpoints** | `lambda/index.ts` | No throttling on `/api/auth/login` or `/api/auth/register`. Brute-force attacks are possible. Add Lambda reserved concurrency or API Gateway with throttling. |
| S8 | **No encryption-at-rest on documents bucket** | `terraform/s3.tf` | S3 default encryption (SSE-S3) is not explicitly enabled on the documents bucket. AWS defaults to SSE-S3 now, but explicit config is best practice. |

### MEDIUM

| # | Issue | File(s) | Details |
|---|---|---|---|
| S9 | **No S3 bucket versioning** | `terraform/s3.tf` | Documents bucket has no versioning enabled — accidental deletes are permanent. |
| S10 | **Pre-signed download URLs valid for 7 days** | `lambda/handlers/documents.ts` | `URL_EXPIRY = 60 * 60 * 24 * 7`. Very long. Consider 1-4 hours max. |
| S11 | **Presigned upload URL does not enforce max file size** | `lambda/handlers/documents.ts` | The presigned S3 PUT URL doesn't include a `Content-Length` condition — users could upload files larger than 25MB directly to S3. |

## 2. CONFIGURATION / BUG ISSUES

| # | Issue | File(s) | Details |
|---|---|---|---|
| C1 | **Malformed IAM policy ARN** | `terraform/iam.tf` | `arn:aws:iam::aws:policy/service-role/AWSLambdaBasicExecutionRole` — missing account number. Should be `arn:aws:iam::aws:policy/...` is actually fine for AWS managed policies, but double check if Terraform apply fails. |
| C2 | **`lib/dynamodb.ts` and `lib/s3-storage.ts` use hardcoded credentials** | `lib/dynamodb.ts`, `lib/s3-storage.ts` | These frontend-side files reference `AWS_ACCESS_KEY_ID` and `AWS_SECRET_ACCESS_KEY` via `process.env`. Since the app is statically exported, these are never bundled — but these files appear unused (all API calls go through Lambda). **Remove or audit** to avoid confusion. |
| C3 | **Unused AWS SDK deps in frontend `package.json`** | `package.json` | The root `package.json` has `@aws-sdk/client-dynamodb`, `@aws-sdk/client-s3`, etc. These are only needed in `lambda/`. They inflate the frontend build. |
| C4 | **`bcryptjs` in frontend deps** | `package.json` | `bcryptjs` is a server-side dependency. Only needed in `lambda/package.json`. |
| C5 | **`next-env.d.ts` committed** | Root | This file is auto-generated by Next.js and should be in `.gitignore`. |

## 3. DEPLOYMENT PRE-REQUISITES

Before deploying, you need to complete these steps **in order**:

### Step 1: AWS Account & CLI

- [ ] Install AWS CLI and configure credentials: `aws configure`
- [ ] Ensure your IAM user has permissions for: S3, DynamoDB, Lambda, IAM, CloudWatch Logs

### Step 2: Terraform State Bucket

- [ ] Create the S3 state bucket manually (one-time):
  ```bash
  aws s3api create-bucket --bucket terraform-state-geekyrbhalala --region us-east-1
  ```

### Step 3: Create `terraform.tfvars`

- [ ] Create `terraform/terraform.tfvars`:
  ```hcl
  jwt_secret = "your-secret-at-least-32-characters-long-here"
  ```
- [ ] Add `*.tfvars` to `.gitignore` (currently missing!)

### Step 4: Run Terraform

```bash
cd terraform
terraform init
terraform plan
terraform apply
```

- [ ] Note the outputs — you'll need them for the next steps.

### Step 5: GitHub Repository Secrets

After `terraform apply`, set these secrets in your GitHub repo (**Settings → Secrets → Actions**):

| Secret Name | Source |
|---|---|
| `AWS_ACCESS_KEY_ID` | `terraform output github_actions_access_key_id` |
| `AWS_SECRET_ACCESS_KEY` | `terraform output -raw github_actions_secret_access_key` |
| `AWS_REGION` | `us-east-1` |
| `NEXT_PUBLIC_API_URL` | `terraform output lambda_function_url` |
| `WEBSITE_BUCKET` | `terraform output documents_bucket_name` → actually use `tax4sure-website` |
| `LAMBDA_ARTIFACTS_BUCKET` | `tax4sure-lambda-artifacts` |

### Step 6: Lambda Dependencies

- [ ] Ensure `lambda/package-lock.json` exists (it does ✓)
- [ ] The CI workflow uses `npm ci` — this requires `package-lock.json`

### Step 7: Verify `.env.local` (local dev only)

- [ ] Copy `.env.local.example` → `.env.local`
- [ ] Fill in `NEXT_PUBLIC_API_URL` with the Lambda Function URL from Terraform output

## 6. RECOMMENDATIONS SUMMARY

**Must-Do Before Deploy:**
1. Create Terraform state bucket
2. Create `terraform.tfvars` with `jwt_secret`
3. Run `terraform apply`
4. Set all 6 GitHub Secrets listed above
5. Add `*.tfvars` to `.gitignore`

**Should-Do (Security Hardening):**
1. Restrict CORS `allow_origins` on Lambda URL and S3 to your website domain
2. Reduce pre-signed URL expiry from 7 days to a few hours
3. Add rate limiting to auth endpoints
4. Enable S3 versioning on documents bucket
5. Switch from IAM access keys to GitHub OIDC for CI/CD
6. Add presigned upload size limit via S3 conditions

1. Remove unused AWS SDK and bcryptjs deps from root `package.json`
2. Remove or mark `lib/dynamodb.ts` and `lib/s3-storage.ts` as deprecated (all calls go via Lambda now)
